# 0001 — Clean Architecture

### Контекст

Микросервис каталога товаров содержит нетривиальную доменную логику: валидацию полей, правила формата slug, статусную модель товара. Эта логика должна быть тестируемой независимо от HTTP и PostgreSQL.

Без явного разделения слоёв:

- бизнес-правила размываются по обработчикам и SQL-запросам,
- смена транспорта или хранилища требует переписывания логики,
- unit-тесты невозможны без поднятия сервера или базы данных.

## Решение

Проект организован по принципу чистой архитектуры с четырьмя явными слоями.

**domain** — ядро системы. Содержит тип `Product`, правила валидации и интерфейс `ProductRepository`. Не импортирует ничего внешнего кроме `lib-validation`. Не знает об HTTP, PostgreSQL и конкретных фреймворках.

**usecase** — оркестрация доменных операций. Каждый use case реализует единый обобщённый интерфейс `Executer[C, R]` и работает только с `domain.ProductRepository`. Не знает о транспорте.

**handler** — транспортный слой. Декодирует HTTP-запрос, вызывает use case, формирует ответ. Не содержит бизнес-логики.

**dbstorage** — реализует `domain.ProductRepository` поверх sqlc. Маппит ошибки хранилища в доменные типы.

Зависимости направлены строго внутрь:

```
handler → usecase → domain ← dbstorage
```

### Последствия

- Доменная логика тестируется без HTTP и БД.
- Use cases тестируются через мок-репозиторий.
- HTTP-обработчики тестируются через мок use case.
- Смена хранилища или транспорта не затрагивает domain и usecase.

## Ограничения

Дополнительная структура увеличивает количество файлов и типов. Для тривиальных CRUD-операций без логики этот overhead может быть избыточен.
