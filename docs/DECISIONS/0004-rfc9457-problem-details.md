# 0004 — RFC 9457 Problem Details для HTTP-ответов

### Контекст

API должен возвращать ошибки в машиночитаемом формате, который клиент может обрабатывать программно, не разбирая строки сообщений. Нестандартные форматы ошибок требуют дополнительной документации и усложняют интеграцию.

## Решение

Все HTTP-ответы — успешные и ошибочные — используют единый JSON-конверт на основе [RFC 9457](https://www.rfc-editor.org/rfc/rfc9457):

```json
{
    "type":       "about:blank",
    "status":     422,
    "title":      "Unprocessable Entity",
    "detail":     "invalid order",
    "instance":   "/api/v1/products",
    "meta":       { "request_id": "abc123" },
    "extensions": { "fields": [ ... ] }
}
```

Успешные ответы несут поле `data`. Ошибочные — `detail` и опционально `extensions.fields` с перечнем ошибок по полям. `Content-Type` для ошибок — `application/problem+json`.

Поле `type` всегда `about:blank` — статус-код является первичной классификацией ошибки.

### Последствия

- Клиент получает стабильный формат для всех ошибок.
- Ошибки валидации содержат исчерпывающий список проблемных полей за один запрос.
- `KindInternal` никогда не раскрывает `detail` клиенту.

## Ограничения

Единый конверт для успешных и ошибочных ответов нетипичен — большинство API возвращают данные напрямую без обёртки. Это требует явного соглашения с клиентами.
