# 0005 — PostgreSQL как хранилище

### Контекст

Сервис каталога товаров требует надёжного реляционного хранилища с поддержкой уникальных индексов, мягкого удаления и сортировки по нескольким полям.

## Решение

PostgreSQL 18 — единственная внешняя зависимость сервиса.

Выбор обоснован:

- нативная поддержка UUID v7 через расширение `pg_uuidv7` (временно-упорядоченные идентификаторы без внешней генерации),
- partial unique index на `slug WHERE deleted_at IS NULL` обеспечивает уникальность slug среди неудалённых товаров без дополнительной логики в коде,
- `TIMESTAMPTZ` для временных меток обеспечивает корректное хранение с учётом часового пояса,
- `product_status` как нативный enum — база отклонит невалидное значение статуса на уровне хранилища.

Схема управляется декларативно через `migrations/current_schema.sql`. Миграции генерируются Atlas и применяются через Goose.

### Последствия

- Partial unique index исключает коллизии slug при мягком удалении без усложнения логики в коде.
- Enum `product_status` в БД дублирует `domain.ProductStatus` в Go — при добавлении нового статуса нужно обновить оба места.
- Сервис stateless — горизонтальное масштабирование не требует координации между инстансами.

## Ограничения

Сервис не использует кэш и не публикует события. Все операции чтения идут напрямую в БД.
