# 0008 — Atlas + Goose для миграций

### Контекст

Управление схемой БД требует воспроизводимых, проверяемых миграций. Основные подходы: ручное написание UP/DOWN SQL, декларативное управление схемой с автогенерацией diff, или комбинация обоих.

## Решение

Используется комбинация [Atlas](https://atlasgo.io) и [Goose](https://github.com/pressly/goose).

**Atlas** управляет схемой декларативно. Актуальное состояние схемы описывается в `migrations/current_schema.sql`. При необходимости изменить схему разработчик обновляет этот файл и запускает `make mig-diff` — Atlas вычисляет diff между текущим состоянием и желаемым и генерирует миграцию.

**Goose** применяет миграции в формате `+goose Up` / `+goose Down`. Atlas генерирует файлы в этом формате.

Схема в `migrations/current_schema.sql` используется также в интеграционных тестах для инициализации тестового контейнера — это единственный источник правды для структуры БД.

### Последствия

- Разработчик описывает желаемое состояние схемы, а не шаги перехода — Atlas вычисляет diff автоматически.
- `atlas migrate lint` в CI проверяет деструктивные изменения до применения.
- `current_schema.sql` одновременно используется для тестов и как документация актуальной схемы.

## Ограничения

Atlas требует dev-базы для вычисления diff (`--dev-url`). В CI это решается через `docker://postgres`.

Goose-комментарии (`-- +goose Up`) в сгенерированных файлах означают, что файлы нельзя применять напрямую через psql без Goose.
