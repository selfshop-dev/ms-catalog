# 0009 — ko для сборки контейнерных образов

### Контекст

Стандартный путь сборки Go-сервиса в контейнер — написать Dockerfile с многоэтапной сборкой. Это работает, но добавляет файл который нужно поддерживать: версия Go, базовый образ, флаги сборки, копирование бинаря.

## Решение

Используется [ko](https://ko.build) — инструмент сборки контейнерных образов для Go без Dockerfile.

ko самостоятельно:

- определяет версию Go из `go.mod`,
- собирает бинарь с правильными флагами (`CGO_ENABLED=0`),
- упаковывает его в минимальный distroless-образ,
- публикует в реестр или загружает локально (`-L` для локального docker daemon).

Сборка prod-образа:

```bash
ko build ./cmd/ms-catalog -B -L
```

Build-метаданные (версия, commit hash, дата) передаются через ldflags в `cmd/ms-catalog/values.go`.

### Последствия

- Отсутствие Dockerfile снижает операционную сложность.
- Distroless-образ не содержит shell и системных утилит — меньше поверхность атаки.
- Образ воспроизводим: ko использует фиксированные базовые образы.

## Ограничения

ko не подходит для сервисов с CGO-зависимостями или нестандартными системными зависимостями. Для dev-окружения с hot-reload используется отдельный `Dockerfile.hot-reload` с air — ko здесь не применяется.
