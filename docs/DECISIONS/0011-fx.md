# 0011 — uber/fx как DI-контейнер

### Контекст

По мере роста сервиса количество зависимостей между компонентами (конфиг → логгер → пул БД → репозитории → use cases → хендлеры → сервер) увеличивается. Ручная сборка в `main` превращается в длинную последовательность инициализации с явным управлением порядком и lifecycle.

## Решение

Используется [uber-go/fx](https://github.com/uber-go/fx) для сборки графа зависимостей и управления lifecycle компонентов.

Каждый компонент описывает свои зависимости через параметры конструктора — fx разрешает граф автоматически. Компоненты с состоянием регистрируют `fx.StartStopHook`:

```go
lc.Append(fx.StartStopHook(e.Start, e.Grace))
```

Порядок старта: конфигурация → логгер → пул БД → роутер → HTTP-сервер.  
Порядок graceful shutdown: HTTP-сервер (drain) → пул БД (close).

Логирование fx-событий направлено в unsamp-логгер на уровне Warn — стандартный вывод fx слишком подробен для production.

### Последствия

- Добавление нового компонента не требует изменений в остальных — только новый `fx.Provide`.
- Lifecycle компонентов управляется централизованно, graceful shutdown гарантирован.
- `fx.Annotate` с тегами `group` используется для регистрации health-чекеров без явного перечисления.

## Ограничения

fx использует рефлексию и добавляет накладные расходы на старт (измеримые только при большом количестве компонентов). Ошибки в графе зависимостей обнаруживаются в рантайме, а не на этапе компиляции.
