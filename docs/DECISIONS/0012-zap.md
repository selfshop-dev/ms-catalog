# 0012 — zap для структурированного логирования

### Контекст

Production-сервис требует структурированного логирования: машиночитаемый формат (JSON) для агрегации в ELK/Loki, типизированные поля для фильтрации, минимальные аллокации на hot path.

Стандартный `log/slog` появился в Go 1.21 и является разумной альтернативой. Существующая экосистема selfshop-dev построена на zap.

## Решение

Используется [uber-go/zap](https://go.uber.org/zap) через обёртку `lib-logger`.

`lib-logger` реализует dual-core архитектуру:

- **sampled core** — обрабатывает сообщения ниже порогового уровня с token-bucket сэмплингом,
- **critical core** — никогда не сэмплируется, доступен через `Logger.Unsampled()` для аудит-логов.

Структурированные поля запроса (trace ID, request ID, user ID) инжектируются через `WithContext` и `FieldExtractor` из пакета `extractors`.

Runtime-управление уровнем логирования доступно через `PUT /log/level` без перезапуска сервиса.

В dev-режиме используется консольный энкодер с цветовой разметкой уровней. В prod — JSON.

### Последствия

- Аудит-события (смена уровня логирования) никогда не теряются из-за сэмплирования.
- Уровень логирования можно изменить в production без деплоя.
- `Logger.With` возвращает новый экземпляр — receiver не мутируется, безопасно в конкурентной среде.

## Ограничения

zap не совместим с `log/slog` на уровне интерфейса. Переход на slog в будущем потребует замены всех мест использования.
