# 0014 — testcontainers для интеграционных тестов

### Контекст

Интеграционные тесты слоя dbstorage и e2e-тесты требуют реальной PostgreSQL. Альтернативы:

- общая dev-база — тесты зависят от внешнего состояния, нельзя запускать параллельно в CI,
- SQLite / in-memory — не поддерживает PostgreSQL-специфику (enum, partial index, `uuidv7`),
- docker-compose в CI — сложная синхронизация старта контейнера и тестов,
- testcontainers — программный запуск контейнеров из тестового кода.

## Решение

Используется [testcontainers-go](https://golang.testcontainers.org) для запуска `postgres:18-alpine` непосредственно из тестов.

Контейнер запускается один раз на весь тестовый бинарь через `sync.Once` в `pkg/db/dbtest/get_pool.go`. После старта применяется `current_schema.sql` из `migrations/current_schema.sql`. Пул прогревается через `Ping` до возврата из инициализатора — первый запрос не тратит время на установку соединения.

Изоляция между тестами обеспечивается на уровне транзакций, а не контейнеров — см. [0015](0015-transaction-isolation.md).

### Последствия

- Тесты работают против реального PostgreSQL с теми же типами, индексами и constraints что в production.
- CI не требует предварительно запущенной базы — контейнер поднимается автоматически.
- Один контейнер на бинарь: повторный запуск тестов не поднимает новый контейнер.

## Ограничения

Первый запуск медленнее из-за pull образа (кэшируется Docker). Тесты требуют доступа к Docker daemon — в некоторых CI-окружениях это требует дополнительной настройки.

Связанные решения: [0015 — Транзакционная изоляция тестов](0015-transaction-isolation.md).
