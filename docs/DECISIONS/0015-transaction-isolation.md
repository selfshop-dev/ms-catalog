# 0015 — Транзакционная изоляция тестов

### Контекст

Интеграционные тесты dbstorage работают против общей базы данных в одном контейнере. Без изоляции данные одного теста влияют на другие. Классические подходы к изоляции:

- `TRUNCATE` перед каждым тестом — медленно, требует знания всех таблиц,
- отдельная база на каждый тест — дорого по времени старта,
- транзакция с откатом — каждый тест работает внутри транзакции, которая откатывается в `TearDownTest`.

## Решение

Каждый тест `dbtest.Suite` получает собственную транзакцию в `SetupTest` и откатывает её в `TearDownTest`:

```go
func (s *Suite[Q]) SetupTest() {
    tx, err := s.Pool.Begin(context.Background())
    s.Require().NoError(err)
    s.tx = tx
    s.ctx = ctxval.With(context.Background(), s.q(tx))
}

func (s *Suite[Q]) TearDownTest() {
    _ = s.tx.Rollback(context.Background())
}
```

Адаптер репозитория получает tx-объект из контекста через `ctxval.Or(ctx, a.g)` — тот же механизм, что используется в production для транзакционных операций. Производственный код не знает что участвует в тестовой транзакции.

### Последствия

- Каждый тест стартует с чистым состоянием без TRUNCATE.
- Откат транзакции быстрее полной очистки таблиц.
- Тесты можно запускать в любом порядке без взаимного влияния.
- Механизм передачи tx через контекст тестируется как side-effect интеграционных тестов.

## Ограничения

Транзакционная изоляция не работает для операций которые выполняются в отдельных транзакциях внутри метода (например, при тестировании retry-логики). Тесты не могут проверять поведение при конкурентных транзакциях.
